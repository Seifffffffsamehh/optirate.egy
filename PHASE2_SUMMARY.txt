5. CACHING STRATEGY
================================================================================

  Implementation: Thread-safe in-memory dict with TTL (services/cache.py)

  Engine              Cache Key Pattern      TTL
  ─────────────────────────────────────────────────────────────────────
  Currency engine     currency_{CODE}        5 minutes (300s)
  Gold engine         gold_all               5 minutes (300s)
  Silver engine       silver                 5 minutes (300s)
  History engine      history_{CODE}_{LIM}   10 minutes (600s)

  Rationale:
    - 5-minute TTL balances freshness vs. upstream load
    - Per-currency caching avoids re-fetching unchanged data
    - Thread-safe locks prevent race conditions under concurrent requests


6. FALLBACK LOGIC
================================================================================

  All three live engines implement a fallback system:

  Currency Engine:
    - If egrates.com scrape returns 0 results for a currency → use static
      fallback rates (snapshot of recent real data)
    - Fallback entries are tagged with "_fallback": true

  Gold Engine:
    - If edahabapp.com scrape returns 0 results → use static fallback
      (24K, 21K, 18K with approximate EGP values)
    - Fallback entries tagged with "_fallback": true

  Silver Engine:
    - If dahabmasr.com scrape returns 0 results → use static fallback
      (approximate per-gram EGP value)
    - Fallback entry tagged with "_fallback": true

  All fallback data uses current timestamps to avoid confusing consumers.

9. ISSUES ENCOUNTERED
================================================================================

  a) Python 3.15 Alpha Compatibility:
     - lxml could not be compiled (requires MSVC C++ Build Tools)
     - Resolved by using Python's built-in html.parser with BeautifulSoup

  b) JavaScript-Rendered Pages:
     - egrates.com per-currency pages initially appeared JS-rendered
     - Investigation revealed the data IS present in server-side HTML tables
     - edahabapp.com and dahabmasr.com also partially render via JS, but
       gold karat text and silver price data are present in the static HTML

  c) egrates.com Scraper:
     - Initial implementation used CSS selectors for /history/ and /banks/ links
     - These links were NOT present in the static HTML (JS-injected hrefs)
     - Rewrote to parse <table><tr><td> structure which IS server-rendered
     - Each row: [bank_name, buy_rate, sell_rate, timestamp]

  d) CBE Historical Data:
     - The CBE historical page may require JS rendering or specific form
       submissions to return data. The endpoint is ready for ingestion
       but may need a scheduled task or manual trigger.
